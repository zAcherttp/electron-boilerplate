Architecture plan · v0.1

Our Electron boilerplate.

An incremental plan for a secure, understandable desktop foundation built with Electron, Vite, React, strict TypeScript, and a Hono Application API. The base stays small; product-specific capabilities remain optional until a real requirement activates them.

01 · Direction

Small base, strong boundaries.

This boilerplate should demonstrate the difficult boundaries once, then get out of the product's way. It is a starting point rather than a framework we must continually fight.

01

Transparent tooling

Vite is the single command entrypoint. Doubleshot integrates the Electron main and preload builds, while electron-builder will package the eventual result.

02

Validate at boundaries

Renderer input, IPC payloads, HTTP requests, environment variables, and external API responses are untrusted. Parse them once at the edge; internal code receives trusted values.

03

Feature ownership

Product behavior lives with its feature. Avoid broad utils, services, helpers, and common buckets that hide ownership.

Non-goal: ship every desktop capability in the base. Database, updater, auth, telemetry, tray behavior, and similar concerns are documented as opt-in modules.
02 · Runtime model

Hono as the Application API.

Hono provides a portable Application API whose current contract is tailored to the desktop application. In embedded mode it receives in-memory Web Standard requests—there is no localhost port. A thin standalone adapter can expose the exact same Hono app over HTTP when needed.

Embedded mode · default

Electron main imports the Hono application factory and calls app.request(). This avoids port discovery, CORS, loopback authentication, and a second process during the initial implementation.

Standalone mode · later

After a web client becomes a real second consumer, a tiny Node adapter can use @hono/node-server. The API itself remains independent of Electron.

Isolation rule: CPU-heavy, crash-prone, or long-running work does not belong in Electron main or the thin Application API. Add a utility process or worker when such work appears.
03 · Ownership

Repository shape.

Keep the Hono application app-local while Electron is its only consumer. Extract packages only after a standalone web runtime becomes a real second consumer.

Allowed dependency direction

renderer ──→ contracts
preload  ──→ contracts
main     ──→ api ──→ contracts

future api-server ──→ extracted api package

Boundaries we will enforce

  • Packages never import from apps.
  • Renderer never imports Electron or Application API implementations.
  • Preload never exposes raw ipcRenderer.
  • Contracts contain schemas and data shapes, not workflows.
  • No root mega-barrels or cross-package relative imports.
04 · Decisions

Technology choices.

Exact versions will be selected from current stable releases during implementation and pinned through the lockfile. Prereleases require an explicit decision.

Concern Choice Reason

Renderer Vite + React Focused use of Vite for the browser runtime and fast development feedback.

Logging Pino Structured logs shared by Electron main and Hono.

06 · Delivery

Incremental implementation.

Each phase should leave the repository runnable and reviewable. We do not begin the next phase until the current acceptance criteria pass.

  1. 00

    Plan and baseline

    • Record architecture, module scope, and dependency direction.
    • Confirm the existing minimal Electron app still starts before migration.
    • Capture the initial git state without rewriting repository history.

    Accept when: this plan is reviewed and the baseline launch is known.

  2. 01

    Workspace and build foundation

    • Move from npm/plain JavaScript to pnpm and strict TypeScript.
    • Keep the API app-local; extract API and server packages only with a second runtime.
    • Configure one Vite entrypoint for renderer, main, and preload builds.
    • Use Doubleshot for Electron startup and development restarts.

    Accept when: install, dev, typecheck, and production build succeed from a clean checkout.

  3. 02

    Secure Electron shell

    • Implement application lifecycle and window ownership.
    • Enable sandboxing and context isolation; keep Node integration disabled.
    • Deny unexpected navigation and new-window requests.
    • Add restrictive development and production content security policies.

    Accept when: the React renderer launches in development and packaged modes without direct Node access.

  4. 03

    Contracts, typed IPC, and Hono Application API

    • Create structured success and error contracts.
    • Add a narrow preload bridge and validate IPC senders and payloads.
    • Create the Hono app factory and in-memory Electron adapter.
    • Add the standalone Node server adapter only with a real web consumer.
    • Deliver the first vertical slice: system.getInfo.

    Accept when: React renders validated system information through preload → IPC → Hono, and the same route passes a direct in-memory request test.

  5. 04

    Renderer application foundation

    • Add typed routes, query ownership, and global error handling.
    • Create a restrained app shell with accessible loading, error, and empty states.
    • Keep product UI under feature owners; shared primitives stay genuinely generic.
    • Do not add a global client-state library until a real state owner requires it.

    Accept when: the vertical slice is navigable, keyboard-accessible, and handles success and failure visibly.

  6. 05

    Testing, quality, and CI

    • Add schema contract tests and Hono request-level integration tests.
    • Add renderer behavior tests and an Electron launch smoke test.
    • Configure format, lint, typecheck, test, and build commands.
    • Run the complete quality gate in GitHub Actions.

    Accept when: CI proves a clean checkout can install, validate, test, and build.

  7. 06

    Packaging and release readiness

    • Completed early: electron-builder, ASAR, isolated release output, unpacked launch, and unsigned NSIS generation.
    • Add product icons and final application identity.
    • Add signing-ready environment configuration without committing secrets.
    • Add package smoke tests and document platform-specific release steps.
    • Prepare release automation; do not publish until explicitly requested.

    Accept when: a locally packaged artifact installs or launches and completes the system-info vertical slice.

07 · Opt-in catalog

Optional modules.

Optional modules are recipes with clear activation triggers. They are not installed in the base merely because many desktop applications eventually need them.

Module Add when Likely implementation Tier
Local database The desktop owns durable structured data or must work offline. SQLite + Drizzle, generated migrations, packaged native-module verification. Common
Secure secrets The application stores access tokens or private credentials. Electron safeStorage or an evaluated OS keychain adapter. Common
Auto updater Signed releases have a stable artifact feed and rollback policy. electron-updater, staged rollout, visible update state, recovery path. Common
Deep links Web pages or other apps need to open a specific desktop route. Custom protocol, single-instance forwarding, strict URL parsing. Common
Native menus and tray The product has background behavior or native commands. Feature-owned command registry shared by menu, shortcut, and UI adapters. Product
File-system access A concrete import, export, workspace, or document workflow exists. Narrow dialog and file APIs with validated paths and explicit permissions. Product
Authentication The Application API calls user-scoped remote services. System browser login, PKCE, callback deep link, secure token storage. Product
Realtime events Polling is insufficient for a defined product workflow. SSE or WebSocket at the API edge, translated into a narrow preload event API. Product
Background work Tasks are CPU-heavy, long-running, restartable, or crash-prone. Electron utility process or worker with explicit job and progress contracts. Product
Telemetry There is a privacy policy, consent model, and operational owner. OpenTelemetry and/or error reporting with sensitive-field redaction. Operations
Internationalization A second locale is committed, not merely possible. Typed message catalogs, locale negotiation, formatting tests. Product
Design system Multiple product screens require reusable visual primitives. Tailwind plus accessible primitives and documented design tokens. Product
Offline synchronization Local and remote sources both own mutable data. Explicit sync state machine, conflict policy, durable operation log. Advanced
Activation rule: before adding an optional module, write its owner, threat model, persistence needs, packaging impact, and acceptance test. Then implement one vertical slice rather than installing an ecosystem all at once.
08 · Guardrails

Definition of done.

A phase is complete when the behavior is observable at its real boundary—not when a set of internal mocks confirms that helper functions called one another.

Test lanes

  • Contract tests for schemas and serialization.
  • Hono integration tests through app.request().
  • Renderer tests for visible behavior and accessibility.
  • Electron end-to-end smoke tests for runtime wiring.
  • Package smoke tests for production paths and assets.

Security invariants

  • Context isolation and renderer sandbox remain enabled.
  • Node integration remains disabled.
  • Raw Electron APIs never cross the preload bridge.
  • IPC senders and untrusted payloads are validated.
  • Renderer bundles contain no server secrets.
  • Navigation and external URLs use explicit allowlists.

Repository gate

Change discipline

  • One implementation phase at a time.
  • Review dependency changes before installation.
  • Keep generated artifacts out of source control unless intentional.
  • Update this plan when architecture decisions change.
  • Commit only after the phase acceptance criteria pass.

This is a living plan. The portable in-memory Hono Application API and its first validated IPC vertical slice are implemented; the build and packaging foundations are verified.
